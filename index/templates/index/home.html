OneCare Developers Portal

The OneCare Developers Portal serves as a comprehensive guide to Python coding standards, designed to elevate the quality and consistency of code across OneCare project. These standards embody industry best practices, emphasizing readability, maintainability, and collaboration among developers. By adhering to these guidelines, developers contribute to a codebase that is not only robust and error-resistant but also scalable for future enhancements and modifications.

Creating a Project

  • Name your directory same as your project name.
  • Initialize virtual environment for each project.
  • Include a README.md file for project details.
  • Maintain a requirements.txt file for Python dependencies (use pip freeze > requirements.txt).
  • Initialize git before starting coding; update changes in git.

Directory Structure

  • Each service with similar modules should have its own directory (e.g., methods, mappings, models, utils).
  • Directories should be named in lowercase.
  • Use a single word for naming directories wherever possible.

Starting a Project

  • Have only one entry point: app.py or main.py.
  • Entry file should be minimal; contains a main() function with no parameters.
  • main() initializes classes and invokes functions/methods.

File Structure

  • Organize modules; group similar functions into modules.
  • Group similar modules into packages.

Imports

  • Differentiate imports as in-built, third-party, and custom.
  • Avoid importing everything using *.
  • Avoid overusing aliasing; use it only when necessary.
  • Mind imports to prevent circular imports.

Spacing & Indentation

  • Use four spaces for indentation; tabs not recommended.
  • Separate import groups with a new line.
  • Separate outer-level objects (classes, functions) with two newline spaces.
  • Separate methods/functions in classes with one newline space.

Variables

  • Avoid modifying global variables.
  • Global variables (CONSTANTS) should be uppercase.

Naming Conventions

  • Use snake_case for variables, PascalCase for classes.
  • Global variables (CONSTANTS) should be uppercase.

Length

  • Limit each line to 80 characters, including newline characters.
  • If code exceeds the line, use a backslash or parentheses to continue.

Conditions

  • No need to explicitly check if a condition is True.
  • Limit nesting to two levels.
  • Use Guard Clause technique to reduce nesting.

Loops

  • Avoid nested loops.
  • Use break or continue appropriately.
  • Avoid nested comprehensions; use generators.

Cohesion and Coupling

  • Ensure functions are highly coherent (do one thing).
  • Reduce coupling; minimize dependencies.

Exceptions

  • Avoid catching generic exceptions.
  • Catch specific exceptions; handle separately.
  • Use custom exceptions with meaningful messages.

Docstring, Comments, and Type Hints

  • Add docstrings for all functions and classes.
  • Add comments sparingly.
  • Use type hints for parameters and returns.

Git

  • Create a branch for each tasks.
  • Avoid working on the main branch.
  • Include initials in branch names.
  • Pull before commit or merge.
  • Prefer cherry-pick over merge.
  • Push code daily.

Testing Standards

Test Organization

  • Maintain a "tests" directory.
  • Organize tests by API endpoint structure.
  • Test should be named after its testing entity
  • i.e, (test_book_model -> BookModel) or (test_author_serializer -> AuthorSerializer)

Test Cases

  • Name test functions descriptively in snake_case.
  • Include positive and negative tests.
  • Consider edge cases and extreme input values.

Test Setup

  • Use mocking for external dependencies.
  • Configure a separate testing environment or use fixtures.

Setup and Teardown

  • Use fixtures for setup and teardown procedures.
  • Leverage fixtures for a consistent test environment.

Test Execution

  • Automate tests using frameworks like pytest.
  • Ensure test cases can run independently.
  • Include relevant assertions for response validation.
  • Log execution details for debugging.

Data Management

  • Manage test data separately from production data.
  • Implement cleanup mechanisms after test execution.

Reporting

  • Generate test reports for each run.
  • Document the purpose and outcome of each test.

Continuous Integration

  • Integrate API tests into CI/CD pipelines.
  • Set up notifications for test failures.

Security

  • Include tests for input validation.
  • Validate proper authorization mechanisms.

Performance Testing

  • Consider load testing for varying loads.
  • Measure and validate response times.

Versioning

  • Test compatibility with different API versions.
  • Include tests for deprecated endpoints.

Documentation

  • Document testing strategy for each endpoint.
  • Add comments in the code for clarity.
